description: 設計ドキュメントで使用する全用語の定義
---

> [!TIP]
> この用語集は設計の Single Source of Truth（SSoT）です。
> ドキュメント内で用語の意味が不明な場合はここを参照してください。

---

## 組織・権限

| 用語 | 英語 | 定義 |
|---|---|---|
| テナント | Tenant | 会社/組織の単位。データ分離の最上位境界。すべての業務データは必ず1つのテナントに属する |
| 組織単位 | Org Unit | 部署・チームなど。承認経路や閲覧範囲に影響する（将来対応） |
| ロール | Role | 役割のまとまり（Member / Approver / PM / Accounting / IT Admin / Tenant Admin） |
| 権限 | Permission | 機能単位の許可（閲覧 / 作成 / 承認 / 締め / 出力など） |
| RBAC | RBAC | Role-Based Access Control。ロールに権限を紐づけて操作許可を制御する方式 |

## データアクセス・セキュリティ

| 用語 | 英語 | 定義 |
|---|---|---|
| RLS | Row Level Security | PostgreSQL の行単位アクセス制御。Supabase の認可の要 |
| 所属 | Membership | ユーザーがテナント / 組織 / プロジェクトに所属している状態。RLS の根拠になる |
| 所有 | Ownership | 特定ユーザーに紐づくデータ（本人の申請、本人の工数など） |
| Service Role | Service Role | RLS をバイパスできる強権キー。バッチ処理・管理操作専用。運用範囲を厳しく絞る |

## ワークフロー（申請・承認）

| 用語 | 英語 | 定義 |
|---|---|---|
| 申請 | Request | 承認を必要とする業務データの入れ物（経費、稟議、休暇、工数締め等） |
| ワークフローテンプレ | Workflow Template | 申請の種類（経費申請 / 工数締め / 支払申請など） |
| 承認経路 | Route | 誰がどの順で承認するか（ステップ構造） |
| ステップ | Step | 承認の段階（一次承認→二次承認→経理承認）。現状は単段階 |
| ステータス | Status | 申請のライフサイクル（draft / submitted / approved / rejected / withdrawn / closed） |

## 会計・経費・請求

| 用語 | 英語 | 定義 |
|---|---|---|
| 経費明細 | Expense Item | 交通費 / 備品等の個別明細行。領収書が紐づく |
| 経費申請 | Expense Claim | 経費明細を束ねて承認に出す単位 |
| 締め / ロック | Close / Lock | 会計期間などの確定。締め後の編集ルールが設計の核 |
| 請求書 | Invoice | 顧客へ請求するドキュメント。明細行を持つ |
| 入金 | Payment | 請求書に対する入金記録。未入金 / 一部 / 完了を管理 |

## 工数・案件管理

| 用語 | 英語 | 定義 |
|---|---|---|
| プロジェクト | Project | 案件。工数や請求の対象となる単位 |
| タスク | Task | 作業単位。工数と紐づけて進捗を管理する |
| 工数表 | Timesheet | 日 / 週単位の作業時間記録。締めや承認が発生する |

## 監査・履歴・運用

| 用語 | 英語 | 定義 |
|---|---|---|
| 監査ログ | Audit Log | いつ誰が何をしたかの証跡（業務イベント中心）。検索・監査向け |
| 履歴 | History | 状態遷移・差戻し理由・コメント等。業務の流れを追うためのもの |
| ADR | Architecture Decision Record | 重要な意思決定と根拠を残す記録 |
