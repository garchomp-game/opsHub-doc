## 目的 / In-Out / Related
- **目的**: ドキュメントの一貫性と検索性、AI編集耐性を高める
- **対象範囲**: docs/配下（要件/設計/ADR/オンボーディング）
- **Related**: [START HERE](../START-HERE)

## ファイル種別
- **.md**: 要件定義 / 基本設計 / 詳細設計 / ADR（原則）
- **.mdx**: 手順（セットアップ/運用）や導線が重要なページのみ

## ID規約
- 要件: `REQ-001`（カテゴリが必要なら `REQ-A01` でも可）
- 画面: `SPEC-SCR-001`
- API : `SPEC-API-001`
- 詳細: `DD-DB-001` / `DD-MOD-001` / `DD-SEQ-001`
- ADR : `ADR-0001`

## 品質ゲート（最低限）
- 主要REQに受入条件（Given/When/Then）
- 画面仕様に「権限」「エラー」「監査ログポイント」
- DB設計に「RLS方針」「Index」「マイグレーション方針」
- 重要判断はADR化してリンクする

## 禁止（事故防止）
- 要件ページに実装コード詳細を書かない
- 詳細設計で要件の目的を勝手に増やさない（必要なら要件へ差分提案）

## 用語集（Glossary）

| 用語 | 英語 | 定義 |
|---|---|---|
| テナント | Tenant | OpsHubを利用する組織単位。すべての業務データは必ず1つのテナントに属する |
| メンバー | Member | テナントに所属する一般ユーザー |
| 承認者 | Approver | ワークフローで申請を承認/差戻しする権限を持つユーザー |
| PM | PM | プロジェクトの管理者。案件/タスク/工数を管理する |
| 経理 | Accounting | 経費・請求の確認・承認・集計を行うユーザー |
| IT管理者 | IT Admin | テナント横断のシステム管理者 |
| テナント管理者 | Tenant Admin | テナント内のユーザー・ロール管理を行う管理者 |
| プロジェクト | Project | 案件。タスク・工数・経費・請求の親単位 |
| ワークフロー | Workflow | 申請→承認/差戻しの一連の流れ |
| 申請 | Request | 承認を必要とする業務データの入れ物。経費やTimesheetの「実データ」とは別概念 |
| 経費 | Expense | 業務上の経費精算 |
| 請求 | Invoice | 取引先への請求管理 |
| 締め/ロック | Close / Lock | 会計期間やデータの確定。締め後の編集ルールが設計の核 |
| 監査ログ | Audit Log | いつ誰が何をしたかの証跡（業務イベント中心）。検索・監査向け |
| 履歴 | History | 状態遷移・差戻し理由・コメント等。業務の流れを追うためのもの。監査ログとは別 |
| RLS | RLS | Row Level Security。PostgreSQLの行レベルアクセス制御 |
| RBAC | RBAC | Role-Based Access Control。ロールによるアクセス制御 |
