# Public Docs vs Internal Docs — QA回答（クライアント決定）
日付: 2026-02-23

前提：**Internal Docs（設計SSoT）を正**とし、Public Docsは「ユーザーが実際に触れる挙動」に合わせて表現を調整します。
（Public Docs側で仕様を変える必要がある場合は、ADRで意思決定を残してからInternalへ反映）

---

## 結論（対応方針）
- **即修正（Critical）**：QA-001
- **次回更新で修正（Medium相当）**：QA-002 / QA-003 / QA-009
- **余裕あれば改善（Low）**：QA-005 / QA-006 / QA-008
- **対応不要（表現差異/概要省略）**：QA-004 / QA-007（※ただし注記追加は推奨）

---

## QAごとの回答（決定）

### QA-001: テナント削除の権限が矛盾（Critical）
**回答（決定）**：✅ **テナント削除は IT Admin のみ可能**（Tenant Adminは不可）  
**理由**：権限昇格・誤削除のリスクが高く、運用上「IT管理者に限定」するのが妥当。Internal Docs側（API/SCR）が一貫している。

**Public Docs修正案**
- `admin/tenant-settings.mdx` のテナント削除UI説明に **「この操作はIT管理者のみ」** 注記を追加
- 可能なら **Tenant Admin には削除セクション自体を非表示**（Public Docs上も「表示されません」と記載）
- 削除手順の説明は **IT Admin向けページへ移動**しても良い

---

### QA-002: タスク done→in_progress 遷移が未記載（Medium）
**回答（決定）**：✅ **完了（done）→進行中（in_progress）への再開を許可**  
**理由**：運用で「誤完了」や「やり直し」が普通に起きるため、再開がある方が自然。Internal APIに存在。

**Public Docs修正案**
- 遷移ルール表に `done → in_progress（再開）` を追記
- 表現例：
  - 「完了済みタスクは **再開** できます（進行中に戻す）」

---

### QA-003: Tenant Admin の IT Admin ロール付与制限が未記載（Medium）
**回答（決定）**：✅ **Tenant Admin は IT Admin ロール付与不可**（IT Adminのみ可能）  
**理由**：権限昇格を防ぐセキュリティ要件。Internal SCRに明記されている。

**Public Docs修正案**
- `guides/user-management.mdx` に注記：
  - 「IT管理者ロールの付与は **IT管理者のみ** が行えます」
- UI上の挙動も合わせる（Tenant AdminにはIT Admin選択肢を出さない、またはエラー説明）

---

### QA-004: 対象企業規模の表現差異（Low）
**回答（決定）**：✅ Public Docsの「10〜500名」表現を基準に、Internal側も表現統一  
**理由**：ブランディング/説明の揺れ。機能仕様ではないので、読み手が迷わない方を採る。

**修正案**
- Internal Project Briefを「10〜500名（中小〜中規模）を主対象、拡張で大規模も視野」等に調整

---

### QA-005: ユーザーステータス値が簡略化（Low）
**回答（決定）**：✅ ステータスは **有効 / 招待中 / 無効** の3値を採用  
**理由**：招待フローがあるなら「招待中」は必須概念。運用トラブルも減る。

**Public Docs修正案**
- `guides/user-management.mdx` に「招待中」の説明を追加
  - 例：招待メール送信済み、未登録（パスワード未設定）状態

---

### QA-006: 工数CSVの小数桁数（Low）
**回答（決定）**：✅ **CSVは小数点2桁（例：8.00）** を標準とする  
**理由**：DBが `numeric(4,2)` 想定でサンプルも2桁。集計・他システム取込を考えると2桁が無難。

**Public Docs修正案**
- `guides/timesheet/reports.md` を「小数点2桁」に修正
- もしUI表示は1桁に丸めるなら、以下を追記：
  - 「画面表示は1桁に丸める場合がありますが、CSVは2桁です」

---

### QA-007: テナント設定項目の省略（Low/対応不要寄り）
**回答（決定）**：✅ Public Docsは概要なので **現状の省略は許容**  
**ただし推奨**：設定項目が増える前提なら、詳細ページへの導線を追加して迷子を防ぐ。

**Public Docs改善案（任意）**
- `guides/tenant-management.md` に追記：
  - 「詳細設定（タイムゾーン、会計年度開始、通知メール、承認経路など）は管理者設定ページを参照」

---

### QA-008: ワークフロー状態遷移図が視覚的に不正確（Low）
**回答（決定）**：✅ 送信済み（submitted）からの **取下げ（withdrawn）を可能** とする（テキストと一致）  
**対応**：図のみ修正（テキストは概ね正しいため、優先度は低め）

---

### QA-009: プロジェクト作成時の入力項目差異（Medium）
**回答（決定）**：✅ 作成時ステータスは **自動で planning 固定**（作成後に変更）  
**理由**：作成直後に状態を選ばせるより、運用上「作ってから変更」が自然。Internal API設計と整合。

**Public Docs修正案**
- PJ作成入力項目から「ステータス」を削除、もしくは
  - 「作成時は計画中（planning）で自動設定」と明記

---

## 追加のおすすめ（将来の事故防止）
- この手の矛盾が出やすい領域（権限/状態遷移/入力項目）は、Internal側に **“Public Docsに出す要約（Public-safe summary）”** を置く
- 重大な決定（QA-001/003/009）は ADR に起票して、意思決定の根拠を固定する
